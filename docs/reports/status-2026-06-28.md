# Status — 2026-06-28 (oracle + camera/perception)

A snapshot of where the two active training workstreams stand, the bugs found this
session, and what was changed. Companion to the topic reports
([domain-randomization](domain-randomization.md), [asymmetric-architecture](asymmetric-architecture.md),
[perception](perception.md), [multicar_throughput](multicar_throughput.md), [hpo](../hpo.md)).

## TL;DR

- **Multi-car asym oracle was not learning** — flat ~5% max-progress, ~2-step
  episodes, 100% off-track over ~1M steps. Root cause: the **multi-car path applied
  every DR magnitude at full strength from step 0** (it has no in-loop held-out eval
  to drive feedback-ADR), and an **unobservable ±15° per-episode steering bias** made
  the task an unlearnable POMDP. Fixed with a **DR warmup** + **frame-stack memory**.
- **Two latent `// n_envs` bugs** in the SB3 callback wiring (checkpoints AND eval)
  meant: at n_cars=12 the oracle wouldn't checkpoint until 1.2M steps, and at n_cars=2
  the **camera run's eval never fired at all** (no eval metrics, no path-plot images,
  no eval-phase perception frames). Both fixed (`freq // n_envs`).
- **Oracle HPO built** (`experiments/oracle_hpo.py`), single-car (real held-out
  objective), tuning PPO + frame-stack + net width + feature-noise; winner transfers
  to the multi-car run. Code-only; run later.
- **Single-car held-out eval harness** ready (`scripts/eval_physical_tracks.py` now
  forwards `GYM_DR_FEATURE_SET` + `GYM_DR_ASYM_CRITIC`, so it reconstructs the asym
  Dict obs for a feature/asym model). Needed because multi-car can't `set_world`.
- **Camera/perception run** (`camera_cnn_dataset`) is the priority dataset generator;
  it's mid-sweep and healthy, but the eval gap above + the 2-camera-car limit are
  under a dedicated definitive-fix pass (see "Camera system").

## Oracle (Test 1 — state-based feature policy, upper bound + camera teacher)

### What broke
`experiments/oracle_asym_multicar.py` (12 cars / 12 distinct tracks, asymmetric
critic, feature_noise DR) trained ~1M steps with **no learning**: `ep_max_progress`
flat at ~5.0, `ep_length` ~2.1 steps, `ep_ended_offtrack` = 1.0, `ep_mean_steering_abs`
~21°. The flatness (not just low) ruled out a per-car spawn/offset bug — if one car
were learning, the aggregate would trend up. So the cause was uniform across cars: the
DR.

### Root cause
The multi-car VecEnv reads each DR knob as `spec_bounds(spec)[1]` — the **static high**
— and threads **no ADR controller**. So `feature_noise=0.3`, `steering_noise=3`, and
especially `steering_bias=±15°` were all applied at **full strength from the first
step**. The per-episode steering bias is a *constant lean* that is **unobservable in a
single feature frame and invisible to the privileged critic too** (it lives in action
space). No single steering correction works for both a +15° and a −15° episode, so a
memoryless MLP cannot learn it — the policy was stuck in an unlearnable POMDP, going
off-track within ~2 steps every episode.

### Fix (two complementary changes)
1. **DR warmup** (`gym_dr/envs/multi_car.py`, `_dr_scale()`): a self-counted linear
   ramp ∈ [0,1] over the first `GYM_DR_DR_WARMUP_STEPS` timesteps, multiplying every
   magnitude knob (bias, feature noise, actuator noise). The policy learns to **drive
   first** (near-clean, survivable early episodes), then to **counter the
   perturbations** as they grow to full strength. This is the multi-car substitute for
   feedback-ADR (which needs an in-loop eval signal multi-car doesn't have). 0 = off
   (legacy behaviour); the camera run is unaffected.
2. **Frame-stack memory** (`frame_stack=4` in the oracle): observation-level memory so
   the MLP sees the recent feature trajectory. A constant lean shows up as a **drift
   signature** (sliding off-center despite centering commands); a few stacked frames
   are enough to infer the bias and apply a standing correction — online system-id
   **without a recurrent policy**. Chosen over RecurrentPPO deliberately:
   frame-stacking is just `VecFrameStack` obs preprocessing, so it ports **unchanged**
   to a Lagrangian/safe-RL PPO on Safety-Gymnasium later (there is no recurrent
   Lagrangian PPO to migrate to). Validated sim-free that
   `VecFrameStack(4) + AsymmetricActorCriticPolicy + PPO.learn` works (actor reads the
   stacked `actor` key, critic the stacked `critic` key).

Oracle run params now: 12 cars, 3M steps, warmup = 600k (20%), frame_stack=4,
asymmetric critic, `centerline_quadratic` reward.

### Checkpoint + eval cadence bug (`gym_dr/trainers/sb3/__init__.py`)
SB3 callbacks tick **once per VecEnv step**, not per timestep, so a raw `save_freq`/
`eval_freq` means `freq * n_envs` timesteps. At 12 cars the first checkpoint wouldn't
land until 1.2M steps (the broken run died at 983k with an empty `checkpoints/`). Now
both are divided by the env count (`freq // _n_envs`) so they mean **timesteps**
regardless of car count.

### Held-out generalisation eval (separate, single-car)
Multi-car can't hot-swap tracks (`set_world` is single-car only), so the held-out
clean-completion measurement runs **single-car** against oracle checkpoints over the 8
held-out worlds. `scripts/eval_physical_tracks.py --tracks <held-out> --model <ckpt>`
now forwards `GYM_DR_FEATURE_SET` + `GYM_DR_ASYM_CRITIC` so it rebuilds the matching
Dict{actor,critic} obs for the asym feature policy. `gym_dr/evaluate.py::evaluate_on_tracks`
does the per-track `set_world` loop and aggregates clean_completion / completion /
max_progress / off-track rate.

## Oracle HPO (`experiments/oracle_hpo.py`) — built, run later

Single-car asym oracle base (has a real in-loop held-out clean-completion objective via
the ACL curriculum's `set_world` rotation — the multi-car oracle can't, so it's the
wrong base for HPO). Searches `learning_rate, ent_coef, n_steps, batch_size, gamma,
gae_lambda, clip_range, n_epochs, target_kl, frame_stack ∈ {1,2,4,8}, net width ∈
{64,128,256}, feature_noise ceiling`. Objective = held-out clean-completion (maximise),
Optuna TPE + lenient MedianPruner. Every knob is config-driven, so the winner drops
straight into `oracle_asym_multicar.py`. Short trials by default (240k steps) to rank
configs by early held-out learning; `n_parallel` workers share one SQLite study. A mild
±5° per-episode steering bias is kept in the base so `frame_stack` has an unobservable
quantity to infer (the production run's bias is ±15°).

## Camera system (priority: CNN dataset + image-policy baseline)

`experiments/camera_cnn_dataset.py` (2-car camera PPO + perception-dataset recorder),
mid-sweep, healthy; dataset accumulating to NVMe via `scripts/perception_offload.py`.

Two issues under a dedicated definitive-fix pass:
1. **Eval never ran** (the `eval_freq // n_envs` bug above): at n_cars=2,
   `eval_freq=60k` fired at 120k timesteps but each chunk is 60k → never. So no eval
   metrics, no path-plot images, and the perception dataset had only a `train/` dir
   (no `eval/`). Fixed in the trainer; the live run's per-chunk containers bind-mount
   the repo, so chunk 35+ pick it up. **Caveat:** multi-car can't `set_world`, so when
   eval fires it runs on the current pair, not the 4 reserved held-out tracks — those
   need a separate **single-car** capture pass (same harness as the oracle eval).
2. **2-camera-car limit — it's a LAUNCH/CONFIG limit, not a rendering limit**
   (independent investigation, code/config evidence). `racetrack_with_racecar.launch`
   only ever spawns **2 car bodies** (`racecar_0/1`, hardcoded `<include>` blocks +
   `car_node.py args="2"`); nothing spawns car bodies dynamically (the runtime
   `SpawnModel` calls are for tracks/obstacles/spectator-cameras only). So a 3rd car's
   camera topic `/racecar_2/camera/.../image_rect_color` has **no publisher**, and the
   agent-side blocking read `DoubleBuffer.get(block=True, timeout=120.0)`
   (`sensors_rollout.py`) starves → `log_and_exit` kills the run (the "~120s in"
   symptom). The `multi_car.py` comment "Gazebo Classic renders only 2 camera sensors
   per world" is a **misdiagnosis**; `deepracer_env.launch` itself says ">2 needs more
   car blocks in racetrack_with_racecar.launch (follow-up)". Feature mode "scales to 8"
   only because a missing model's *state* read returns `success=False` without blocking
   (phantom agents that still tick `step()`); camera mode can't because its sensor read
   *blocks*. `docs/reports/multicar_throughput.md`'s old "n=3 render limit" claim
   measured "3 agents, 2 cars" and mislabeled it.
   - **Raisable:** generate `racecar_2..N` `<include>` blocks (looped, not copy-paste) +
     parameterize `car_node.py args` to the real `n_cars`. **But** the *real* residual
     limit then appears: Gazebo Classic serialises all camera rendering through one OGRE
     thread, and n=2 already saturates render (~51 steps/s, effective RTF ~1.0). A real
     3rd 640×480@15 Hz camera adds ~33% render load → **graceful fps/RTF degradation**,
     not a hard failure. Headroom: lower camera `update_rate` (15→10 Hz), shrink
     resolution, lower RTF, GPU rendering — or scale out via **separate Gazebo
     processes** (1–2 camera cars each) / the ROS2-Gz-Sim port. So raising the count
     trades throughput for the *current working* 2-car fidelity; not pursued by default.
   - **BENCHMARK (2026-06-28, CONFIRMED).** Generalised the launch (`racecar_2..7`
     `<include>` blocks gated on the car count + `car_node` args = real `n_cars`,
     backward-compatible: n=2 unchanged) and benchmarked camera n=2/3/4 on the GPU box
     (`rostopic hz` per car + effective RTF + steps/s):

     | n | cameras @15Hz | steps/s | RTF_eff | n×RTF_eff |
     |---|---------------|---------|---------|-----------|
     | 2 | 2/2 | 53 | 2.07 | 4.14 |
     | 3 | 3/3 | 50 | 1.35 | 4.05 |
     | 4 | 4/4 | 46 | 0.99 | 3.96 |

     `racecar_2/_3` publish at full **15 Hz** (were 0 Hz) → the cap was the LAUNCH limit,
     now lifted. The real residual (single OGRE render thread) shows as **graceful RTF
     degradation** (2.07→0.99), not a crash. **n×RTF_eff is flat ~4.0** → adding camera
     cars to ONE world gives NO total-throughput gain (render saturated at n=2). So for
     more camera throughput use **parallel Gazebo processes**, not more cars/world; raise
     the in-world count only when you genuinely need >2 agents together (e.g. head-to-head),
     accepting the lower RTF. The dr-gym n>2 guard stays (override `GYM_DR_ALLOW_CAMERA_NCARS=1`).

### Definitive camera fix (from the correctness review)

The review verified the priority items are **correct**: frame↔target alignment +
per-episode contiguity (valid 4-frame stacking, no off-by-one), the `_n_envs` cadence,
the DR-warmup leaving the camera path untouched, and the multi-car eval running without
crashing. It found 3 real bugs, all in the eval/early-stop path — now fixed:

1. **[HIGH] Held-out eval was mislabeled + held-out frames never captured.** Multi-car
   has no `set_world`, so `MultiWorldEvalCallback` "evaluated" all `EVAL_WORLDS` on the
   *current* training pair → fake per-world metrics + a ~0 `generalization_gap`, and zero
   held-out frames. Fix: `MultiCarVecEnv` now advertises `can_set_world=False`; the
   callback detects it and runs **one honest in-distribution eval on the current tracks**
   (drives early-stop + captures eval-phase frames), and **stops emitting the fake
   per-held-out-world metrics / gap**. True held-out coverage is the separate single-car
   capture pass below.
2. **[MED] Eval→train phase contamination.** The world-restore happened while the
   recorder phase was still `eval`, so the first post-eval *training* episode flushed to
   `eval/`. Fix: restore phase to `train` **before** the resume reset (in `callbacks.py`).
3. **[MED] Mastery early-stop could never fire** (`eval_freq == chunk` → one eval/chunk,
   but each chunk is a fresh container that resets the streak). Fix:
   `eval_freq = CHUNK_STEPS // 4` (~4 rounds/chunk ≥ patience).

The live run's per-chunk containers re-import these files, so chunks 35+ pick up all
three. Minor items left as-is (harmless): recorder chmods the shard root each flush; a
misleading "drop partial episodes" comment; <4-frame episodes written but unused.
